Modified processUrl() function inside QrMenuImportScreen composable.

Per D-07 and RESEARCH §1, LocalAiParser.kt is scrapped — it is non-deterministic (temperature=0.2, topK=40), its output schema lacks categories/prices, and it is not wired into any pipeline layer.

Per D-08, PdfMenuExtractor.kt and WebViewMenuExtractor.kt remain as unchanged input adapters; their string output is fed to Layer 0 (MenuContentPreprocessor).

New pipeline wiring (after content extraction, before upload):
1. Create LLMEnricher wrapping existing LLMApiClient
2. Instantiate MenuParserPipeline with:
   - llmEnricher = enricher
   - mode = ParsingMode.BALANCED
   - confidenceThreshold = 0.5f (per D-15)
   - enrichmentThreshold = 0.65f
3. Determine sourceType from URL pattern:
   - PDF URL → SourceType.PDF
   - HTTP URL → SourceType.HTML
   - Short non-HTTP text → SourceType.QR_DIRECT
   - Long text → SourceType.RAW_TEXT
4. Call pipeline.parse(content, sourceType)
5. Log result via ParseLogger.log(parseResult)

Fallback logic (D-01, D-03, D-15):
- If parseResult.categories.isNotEmpty() AND parseResult.confidence >= 0.5f:
  - Use new pipeline result
  - Status message based on confidence level:
    - >= 0.65: "✅ Trovati N piatti in M categorie."
    - >= 0.50: "⚠ Trovati N piatti (verifica consigliata, confidenza X%)"
- Else (confidence < 0.5 OR 0 dishes):
  - Status: "Parser deterministico insufficiente, uso AI classica…"
  - Run legacy pipeline: LLMApiClient.processMenuText() + MenuParser().parseMenuText() + LLMApiClient.enrichDishes()
  - Status: "✅ Trovati N piatti in M categorie (via AI). Verifica e conferma."

D-02 satisfied: status messages clearly distinguish pipeline source.
